# AGENTS.md

本文件给后续参与 Dayboard 项目的开发 AI 使用。开始任何开发任务前，请先阅读本文件，理解项目目录边界和协作规则。

## 项目定位

Dayboard 是一个面向 Windows 桌面的轻量日程与任务管理软件。目标是把日历/任务面板贴到桌面上，通过连接 Gmail、Outlook 等账号，同步并展示用户日程，支持月视图、周视图、日视图和快捷编辑，体验上接近轻量版滴答清单。

## 目录用途

- `docs/`
  - 产品文档区。
  - 存放 PRD、需求迭代记录、用户场景、功能优先级、关键产品决策、技术选型决策。
  - 涉及需求变更或架构选择时，优先在这里补充记录。

- `assets/design/`
  - 设计素材区。
  - 存放 UI 效果图、交互稿、视觉参考、组件草图、界面截图。
  - 命名建议包含日期和主题，例如 `2026-06-04-week-view-concept.png`。

- `assets/bug/`
  - 测试报错截图区。
  - 存放崩溃截图、UI 异常截图、同步错误截图、测试失败截图。
  - 截图文件名建议包含问题模块和日期，例如 `sync-google-token-error-2026-06-04.png`。

- `assets/reference/`
  - 参考图与灵感收集区。
  - 存放竞品截图、桌面 widget 灵感、任务管理交互参考、日历视图参考。
  - 这里的内容只作为参考，不应直接复制第三方受版权保护的设计或素材。

- `notes/`
  - 学习笔记区。
  - 存放开发中踩过的坑、调研结论、API 接入记录、Windows 桌面能力笔记、Google/Microsoft 日历同步经验。
  - 每次解决复杂问题后，建议写一篇短笔记，方便后续 AI 和人类开发者接续。

- `code/`
  - 独立代码区。
  - 所有应用源码、脚本、测试、配置文件、构建文件都应放在这里。
  - 不要把源码散落到项目根目录、`docs/`、`assets/` 或 `notes/` 中。

## 开发规则

1. 先读文档，再改代码。
   - 开发前优先查看 `docs/` 中的 PRD、决策记录和最近的迭代说明。
   - 如果需求与现有文档冲突，先记录冲突点，再按用户最新指令执行。

2. 代码只放在 `code/`。
   - 新建项目、安装依赖、添加测试、创建脚本时，工作目录应在 `code/` 内。
   - 根目录只放项目级说明文件，例如 `AGENTS.md`、`README.md`、许可证等。

3. 保持文档同步。
   - 改动影响产品行为时，在 `docs/` 记录变更。
   - 解决重要技术问题时，在 `notes/` 记录原因、方案和验证方式。

4. 保持素材分类清晰。
   - UI 设计相关放入 `assets/design/`。
   - 报错和测试截图放入 `assets/bug/`。
   - 灵感和竞品参考放入 `assets/reference/`。

5. 开发 Windows 桌面能力时要谨慎。
   - 涉及开机自启、托盘、桌面贴片、窗口置底/置顶、OAuth、本地凭据存储时，必须优先考虑权限、安全性和用户可控性。
   - 不要默认收集用户日历数据；任何同步、缓存和日志都应尽量最小化。

6. 日历同步要重视边界情况。
   - 特别注意时区、夏令时、重复事件、全天事件、取消事件、跨账号冲突、离线编辑、OAuth token 过期等问题。
   - 修改同步逻辑时，应补充测试或至少记录手动验证步骤。

7. 不要随意重构。
   - 优先延续现有技术栈和目录约定。
   - 只在能减少真实复杂度或解决明确问题时引入新依赖、新框架或新抽象。

8. 输出前要验证。
   - 代码变更后尽量运行可用的 lint、test、build 或启动检查。
   - 如果无法验证，请在最终说明中明确原因和剩余风险。
