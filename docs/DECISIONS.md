# Dayboard 决策记录

日期：2026-06-04

## D001：源码统一放在 code/

状态：已接受

依据：AGENTS.md 已明确规定所有应用源码、脚本、测试、配置文件和构建文件都放在 `code/`。根目录只保留项目级说明文件。

影响：

- 后续初始化项目时工作目录使用 `code/`。
- 不在根目录散落 package.json、Cargo.toml、测试脚本等源码文件。

## D002：MVP 先做本地体验，不直接接真实同步

状态：已接受

依据：日历同步涉及 OAuth、token 安全、时区、夏令时、重复事件、全天事件、取消事件、跨账号冲突和离线编辑。过早接入会拖慢桌面贴片体验验证。

影响：

- 第一阶段使用本地数据或假数据。
- Google/Outlook 同步从 Phase 2 开始。
- PRD 将同步列为后续阶段能力。

## D003：首选技术栈暂定 Tauri v2 + React + TypeScript + SQLite

状态：暂定

依据：

- Tauri v2 官方插件覆盖 autostart、single-instance、sql、store、stronghold、updater、window-state、system-tray 和 window-customization 等 Dayboard 需要的能力。
- Tauri 比 Electron 更适合轻量、本地优先、长期常驻的桌面应用定位。
- React + TypeScript 适合快速构建月/周/日视图和表单交互。
- SQLite 适合本地任务、事件缓存和同步状态。

备选：

- Electron + React：生态成熟，窗口行为 API 完整，但体积和常驻资源占用更高。
- Wails + React + Go：已有相似项目可参考，但托盘、多窗口、updater、credential storage 的工程边界还需进一步核验。
- WinUI 3/.NET：Windows 原生体验更好，但 UI 迭代速度和跨平台空间较弱。

待验证：

- Tauri 在 Windows 上实现桌面贴片层级、透明/半透明、任务栏隐藏、托盘菜单、窗口位置记忆的实际表现。
- Tauri Stronghold 或 Windows Credential Manager/DPAPI 用于 token 存储的可行性。

## D004：token 和敏感数据不得明文长期存储

状态：已接受

依据：Dayboard 会处理用户日历、邮箱账号和 OAuth token。AGENTS.md 要求同步、缓存和日志尽量最小化。

影响：

- MVP 阶段不接真实 token。
- 同步阶段必须先确定安全存储方案。
- 日志默认不得记录事件正文、token、refresh token、邮箱地址。

## D005：开机自启默认关闭

状态：已接受

依据：桌面常驻能力必须用户可控。开机自启属于系统行为，不能默认开启。

影响：

- 设置页提供开关。
- 托盘或首次引导可以提示，但不自动启用。

## D006：主界面优先服务桌面工作板，而不是小工具卡片

状态：已接受

依据：Dayboard 需要贴到桌面上持续查看和编辑任务。若界面像小面板，概览、底部导航和装饰元素会挤占日历空间，任务标题难以辨认。

影响：

- 默认窗口尺寸按接近半屏的桌面面板设计。
- 日历区域是主交互区，直接显示任务/日程标题。
- 透明度、固定方式等低频设置放入设置面板，不占主界面空间。
- 任务列表/详情面板默认可折叠，常态下让日历占满主要空间。
- 点击日历格子的空白区域直接新建任务，减少额外按钮依赖。
- 主界面弱化品牌，不常驻显示产品名；保留设置、托盘和关于页作为品牌出现位置。
- 操作入口使用悬浮工具条，并支持左侧/右侧切换，以适配用户把日历贴在屏幕左侧或右侧的习惯。
- 月/周/日视图切换属于低频操作，放入设置面板，主界面仅保留轻量月份信息和翻页入口。
