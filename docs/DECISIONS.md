# Dayboard 决策记录

日期：2026-06-04

## D001：源码统一放在 code/

状态：已接受

依据：AGENTS.md 已明确规定所有应用源码、脚本、测试、配置文件和构建文件都放在 `code/`。根目录只保留项目级说明文件。

影响：

- 后续初始化项目时工作目录使用 `code/`。
- 不在根目录散落 package.json、Cargo.toml、测试脚本等源码文件。

## D002：MVP 先做本地体验，不直接接真实同步

状态：已接受

依据：日历同步涉及 OAuth、token 安全、时区、夏令时、重复事件、全天事件、取消事件、跨账号冲突和离线编辑。过早接入会拖慢桌面贴片体验验证。

影响：

- 第一阶段使用本地数据或假数据。
- Google/Outlook 同步从 Phase 2 开始。
- PRD 将同步列为后续阶段能力。

## D003：首选技术栈暂定 Tauri v2 + React + TypeScript + SQLite

状态：暂定

依据：

- Tauri v2 官方插件覆盖 autostart、single-instance、sql、store、stronghold、updater、window-state、system-tray 和 window-customization 等 Dayboard 需要的能力。
- Tauri 比 Electron 更适合轻量、本地优先、长期常驻的桌面应用定位。
- React + TypeScript 适合快速构建双周、摘要、月视图和表单交互。
- SQLite 适合本地任务、事件缓存和同步状态。

备选：

- Electron + React：生态成熟，窗口行为 API 完整，但体积和常驻资源占用更高。
- Wails + React + Go：已有相似项目可参考，但托盘、多窗口、updater、credential storage 的工程边界还需进一步核验。
- WinUI 3/.NET：Windows 原生体验更好，但 UI 迭代速度和跨平台空间较弱。

待验证：

- Tauri 在 Windows 上实现桌面贴片层级、透明/半透明、任务栏隐藏、托盘菜单、窗口位置记忆的实际表现。
- Tauri Stronghold 或 Windows Credential Manager/DPAPI 用于 token 存储的可行性。

## D004：token 和敏感数据不得明文长期存储

状态：已接受

依据：Dayboard 会处理用户日历、邮箱账号和 OAuth token。AGENTS.md 要求同步、缓存和日志尽量最小化。

影响：

- MVP 阶段不接真实 token。
- 同步阶段必须先确定安全存储方案。
- 日志默认不得记录事件正文、token、refresh token、邮箱地址。

## D005：开机自启默认关闭

状态：已接受

依据：桌面常驻能力必须用户可控。开机自启属于系统行为，不能默认开启。

影响：

- 设置页提供开关。
- 托盘或首次引导可以提示，但不自动启用。

## D006：主界面优先服务桌面工作板，而不是小工具卡片

状态：已接受

依据：Dayboard 需要贴到桌面上持续查看和编辑任务。若界面像小面板，概览、底部导航和装饰元素会挤占日历空间，任务标题难以辨认。

影响：

- 默认窗口尺寸按接近半屏的桌面面板设计。
- 日历区域是主交互区，直接显示任务/日程标题。
- 透明度、固定方式等低频设置放入设置面板，不占主界面空间。
- 任务列表/详情面板默认可折叠，常态下让日历占满主要空间。
- 点击日历格子的空白区域直接新建任务，减少额外按钮依赖。
- 主界面弱化品牌，不常驻显示产品名；保留设置、托盘和关于页作为品牌出现位置。
- 操作入口使用悬浮工具条，并支持左侧/右侧切换，以适配用户把日历贴在屏幕左侧或右侧的习惯。
- 贴片模式切换属于低频操作，设置里提供默认模式；主界面只保留低显眼度的双周 / 摘要 / 月切换和轻量月份翻页入口。

## D007：OpenDesign 原型作为当前 MVP 的优先视觉参考

状态：已接受

依据：用户确认 OpenDesign 原型的深色、半透明、安静桌面贴片外观更接近目标体验；沟通纠错用截图不作为参考。已废弃的 OpenDesign 页面 `index.html`、`widget-empty.html`、`widget-busy.html` 不纳入项目参考。

影响：

- `assets/reference/opendesign-prototype/` 只保留 `widget-main.html`、设置页、样式和脚本等有用参考文件。
- 当前实现优先深化 OpenDesign 主贴片方向，而不是继续沿用早期普通日历 App 外观。
- MVP 主贴片模式收敛为双周、摘要、月视图；周视图和日视图不再作为当前桌面贴片主入口。
- Gmail / Outlook 在 MVP 中只保留低强调账号入口，不进入真实 OAuth。

## D008：主界面改为 OpenDesign 月视图优先结构

状态：已接受

依据：用户明确反馈旧 React 界面方向不满意，要求以 OpenDesign 导出的原型为基础重新设计。OpenDesign 当前可用原型的核心结构是月视图占主体、右侧当日任务抽屉默认隐藏，并提供月 / 周 / 双周切换。

影响：

- 当前实现从旧的侧边工具条工作板改为 `widget-shell` + `panel--calendar` + `panel--glance-drawer` 结构。
- 默认主视图采用月视图；周和双周作为用户主动切换态。
- 摘要模式暂不作为当前 MVP 主界面入口，后续若需要窄态再单独设计。
- 新建、编辑、完成、拖拽排期继续保留在 OpenDesign 结构内。

## D009：面板按钮必须有语义和反馈

状态：已接受

依据：Dayboard 会作为桌面贴片长期停留。若按钮没有真实作用或没有反馈，用户会把它理解为坏掉的控件。

影响：

- 已实现能力必须直接执行，例如新建、编辑、完成、拖拽排期、视图切换、透明度、主题预览、贴片尺寸。
- 暂未接入真实系统或账号能力的按钮必须给出明确 MVP 阶段反馈，例如 OAuth、开机启动、鼠标穿透、隐藏到托盘的浏览器预览差异。
- 周视图和双周视图的前后导航按周/双周移动，不复用月视图翻月语义。

## D010：拖拽排期不触发任务抽屉展开

状态：已接受

依据：在日历中拖拽任务的用户意图是快速调整日期，而不是查看或编辑目标日期的完整任务列表。若拖拽后自动展开右侧抽屉，会打断日历为主的桌面贴片使用方式。

影响：

- 拖拽任务到新日期后，只更新任务日期并选中目标日期。
- 若右侧任务抽屉原本关闭，拖拽后保持关闭。
- 点击日期、点击任务、点击空白新增仍可按原有语义打开抽屉或编辑器。

## D011：日历源选择进入主贴片，同步改为自动触发

状态：已接受

依据：用户需要在桌面贴片中直接控制同一账号下不同日历的显示范围，并在新建日程时选择具体目标日历；仅在设置页手动同步不符合桌面常驻产品的使用方式。

影响：

- 主贴片左侧提供可折叠日历源面板，按账号列出具体日历并保存显示勾选状态。
- 每个具体日历保留自身颜色，日历条目、当日面板和来源列表使用同一颜色标识。
- 日程数据保存远端日历 ID、显示名称和远端事件 ID，旧数据继续兼容通用来源字段。
- 新建日程时列出所有可写日历，Google 日程写入用户选择的具体日历。
- 编辑已有日程时允许修改目标日历；Google 日历之间使用事件迁移接口，本地与 Google 之间执行对应的创建或删除操作。
- Google 日历在应用启动、窗口重新获得焦点且缓存过期、每 15 分钟以及用户主动刷新时同步。
- 设置页保留同步入口，但与贴片和自动同步共用同一同步流程。

## D012：当前 Google OAuth token 交换必须保留 client secret

状态：已接受

依据：Dayboard 当前使用的 Google OAuth 客户端配置在授权码交换和 refresh token 刷新时需要 `client_secret`。此前安全审核直接移除该参数后，会导致现有 Google 授权配置无法完成 token 交换。

影响：

- `VITE_GOOGLE_CLIENT_SECRET` 必须保留在本地 `.env`，并参与授权码交换与 token 刷新请求。
- 真实密钥不得提交到 Git；仓库只在 `.env.example` 保留变量名和约束说明。
- 后续代码审核不得仅以“客户端不应有 secret”为由直接删除该参数。
- 若要移除或迁移密钥，必须先实现并验证可工作的替代方案，例如由安全后端或合适的原生 OAuth 流程负责 token 交换，然后再更新此决策。
- 当前把 secret 注入前端构建存在可提取风险，这是已知的阶段性限制，不等同于可以在未验证替代方案时删除它。

## D013：日历主视图采用高密度网格

状态：已接受

依据：桌面贴片在固定尺寸内需要尽量多地展示任务。卡片式日期格的大间距、圆角和高任务条显著降低了信息密度，与 Apple Calendar 等成熟日历产品相比，同一空间可见条目过少。

影响：

- 日期格之间使用 1px 分隔线，不再使用独立卡片式间距和大圆角。
- 月、周、双周视图统一压缩格内边距、标题间距和任务列表间距。
- 月视图任务条目标高度约 17px，周与双周约 18px，保留省略号、颜色和键盘焦点反馈。
- 密度优化不得通过隐藏任务标题或降低到不可读字号实现。

## D014：主视图提供回到今天，全天日程必须显式可见

状态：已接受

依据：用户需要在浏览其他月份或周次后快速回到当前日期；创建日程时仅靠“时间留空”隐式推断全天不够清晰，也容易造成误解。

影响：

- 主贴片工具栏提供始终可见的“今天”按钮，点击后同时恢复当前日期和对应月份。
- 日程编辑器提供“全天”开关，开启后清空并禁用开始、结束时间。
- 日程的开始和结束时间都留空时，自动按全天保存；只填写一个时间时阻止保存并提示补全。
- 非全天日程要求结束时间晚于开始时间。
