# 多日历来源与自动同步

日期：2026-07-12

## 方案

- Google `calendarList` 用于发现同一账号下的具体日历。
- 同步保存 Google 日历的 `backgroundColor`，界面条目按具体日历着色，而不是只按账号提供商着色。
- 本地缓存每个日历的 ID、名称、写入权限和显示状态，离线启动时仍可恢复筛选。
- 拉取事件时保留 `calendarId` 与远端事件 ID，创建、更新和删除都使用对应日历路径。
- 编辑日程切换 Google 日历时调用 `events.move`；在本地日历与 Google 日历之间切换时，分别创建或删除对应远端事件。
- Google 不允许对他人组织、当前账号仅持有 attendee copy 的邀请调用 `events.move`。移动请求报错后先查询目标日历：若事件已存在，按远端成功处理并修正本地映射；只有确认目标不存在且错误为 `cannotChangeOrganizer` 时，才提示用户选择复制为独立日程并删除原副本。复制不会保留参会人和邀请关系，用户取消时保持原事件不变。
- 自动同步触发点为应用启动、窗口重新聚焦且距离上次同步超过 5 分钟、每 15 分钟和用户手动刷新。
- 同步使用单一并发锁，避免启动、聚焦和手动刷新同时请求。

## OAuth 注意事项

日历列表读取新增 `calendar.calendarlist.readonly` scope。已有授权如果缺少该 scope，需要断开后重新连接 Google 才能列出全部日历。

当前 Google OAuth 客户端的授权码交换和 refresh token 刷新都需要 `VITE_GOOGLE_CLIENT_SECRET`。真实值只保存在被 Git 忽略的 `code/.env`，但代码中的 `client_secret` 参数不能在安全审核中直接删除。未来只有在安全后端或经过验证的原生 OAuth token 交换替代当前流程后，才能移除该参数。

## 验证重点

- 同一 Google 账号下多个日历能独立勾选显示。
- 新建日程能写入选定的可写日历。
- 全天事件结束日期继续按 Google API 的排他结束日期处理。
- 自动同步失败时保留本地数据并显示可重试状态。
